# Karpathy Guidelines

[Andrej Karpathy의 관찰](https://x.com/karpathy/status/2015883857489522876) 에서 도출한, LLM 코딩 실수를 줄이기 위한 행동 지침.

**트레이드오프:** 속도보다 신중함에 가중치. 사소한 작업에서는 판단으로 생략 가능.

## 1. Think Before Coding

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 표면화할 것.**

구현 전에:
- 가정을 명시적으로 진술한다. 불확실하면 묻는다.
- 해석이 여러 가지면 모두 제시한다 — 조용히 하나를 고르지 않는다.
- 더 단순한 접근이 있으면 말한다. 필요하면 푸시백한다.
- 무엇이 불명확한지 이름을 붙이고 멈춘다.

## 2. Simplicity First

**문제를 푸는 최소 코드. 투기적 코드 금지.**

- 요청되지 않은 기능 추가 금지.
- 일회용 코드에 추상화 금지.
- 요청되지 않은 "유연성" / "설정 가능성" 금지.
- 일어날 수 없는 시나리오에 대한 에러 핸들링 금지.
- 200줄 짠 게 50줄로 가능하면 다시 쓴다.

자문: "시니어 엔지니어가 이걸 과도하다고 할까?" 그렇다면 단순화.

## 3. Surgical Changes

**필요한 것만 건드린다. 자기가 만든 흔적만 치운다.**

기존 코드 편집 시:
- 인접 코드/주석/포맷을 "개선"하지 않는다.
- 부서지지 않은 것을 리팩토링하지 않는다.
- 본인 스타일과 다르더라도 기존 스타일에 맞춘다.
- 무관한 죽은 코드는 언급만 — 삭제하지 않는다.

변경으로 고아가 생기면:
- 본인 변경이 미사용으로 만든 import/변수/함수만 제거.
- 기존부터 죽어 있던 코드는 요청 없이 제거하지 않는다.

테스트: 변경된 모든 줄은 사용자의 요청에 직접 추적 가능해야 한다.

## 4. Goal-Driven Execution

**성공 기준 정의. 검증될 때까지 루프.**

작업을 검증 가능한 목표로 변환:
- "검증 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "재현 테스트 작성 후 통과시키기"
- "X 리팩토링" → "전후 테스트 통과 보장"

여러 단계 작업은 간단한 계획을 진술:
```
1. [단계] → verify: [체크]
2. [단계] → verify: [체크]
3. [단계] → verify: [체크]
```

강한 성공 기준 → 독립적으로 루프 가능. 약한 기준("작동하게 해줘") → 끊임없는 명확화 요청.

---

**이 지침이 효과를 내고 있다는 신호:** diff 에서 불필요한 변경 감소, 과복잡화로 인한 재작성 감소, 명확화 질문이 실수 이후가 아닌 구현 이전에 발생.
