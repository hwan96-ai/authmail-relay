# Design: email-service expansions

## Problem Statement
Every Python team rebuilds the same SMTP-with-templates boilerplate (magic link, OTP, CRLF guard, HTML escape) and then operates it blind — no retry, no metrics, no bounce visibility. `email-service` already solves the boilerplate; the next gap is *operational confidence* once it's running behind real traffic.

## What Makes This Cool
A single `pip install` gives you a hardened email microservice **with built-in observability and at-least-once delivery semantics** — without dragging in Celery, Redis, or a SaaS vendor. The whoa: `/metrics` scrapeable by Prometheus on day one, and a 5xx SMTP blip no longer drops the email on the floor.

## Premises
1. Users are Python orgs running an internal-network deployment (compose / k8s sidecar), not SaaS customers — so the comparison set is "stdlib + glue" not Resend.
2. Operational pain dominates feature pain at this stage. The library does enough; the *service* doesn't tell you when it's failing.
3. Synchronous-send-then-502 is the single biggest reliability gap. Transient SMTP failures are routine and currently bubble straight to the caller.
4. Distribution is currently `pip install git+...` — no PyPI release. That caps adoption and is fixable in one afternoon.

## Approaches Considered

### Approach A: Minimal (recommended)
Add `/metrics` (Prometheus text format) + bounded in-process retry with exponential backoff on SMTP transient failures (`SMTPServerDisconnected`, `SMTPConnectError`, 4xx codes). No new deps for metrics (hand-rolled text exposition), no new processes, no DB. ~150 LOC. Counter labels: `endpoint`, `outcome` (sent/refused/retried/failed). Histogram for send latency.

### Approach B: Ideal
Persistent outbox (SQLite by default, Postgres optional) + background worker thread draining the queue + multi-SMTP failover list with circuit breaker + webhook callback to caller URL on terminal success/failure + `/metrics`. True at-least-once. ~800 LOC, new deps, new failure modes (DB locks, worker liveness, dedup keys). Real value but multiplies surface area.

### Approach C: Creative
LLM-generated templates ("describe the email, get HTML") + sentiment-aware retry timing (delay retries when the recipient domain looks angry). Cool demo, wrong layer — couples a boring infra primitive to a model dependency, breaks the "stdlib only in core" premise, and the value is speculative.

## Recommended Approach
**Approach A.** Reasoning:
- **80/20**: `/metrics` + retry covers the two questions every operator asks ("is it working?" and "did it just lose mail?") without changing the deployment topology.
- **Reversible**: pure in-process additions. No DB migration, no schema, no new container. If retry causes problems, set `EMAIL_RETRY_MAX=0` and it's off.
- **TTHW preserved**: still one `docker compose up`. Approach B doubles the moving parts.
- **Observation-friendly**: success is literally a curl to `/metrics` and a forced SMTP failure showing `email_send_retries_total` increment. Easy to verify, easy to demo.
- Approach B is the natural follow-up *once Approach A is in prod and the metrics show retry isn't enough*. Don't build the queue before the data justifies it.

## Success Criteria
1. `GET /metrics` returns Prometheus exposition format including `email_send_total{endpoint,outcome}`, `email_send_duration_seconds` histogram, and `email_send_retries_total` — verifiable via `curl | promtool check metrics`.
2. Forced SMTP transient failure (Mailpit stopped mid-send) is retried up to `EMAIL_RETRY_MAX` (default 3) with exponential backoff (100ms → 400ms → 1600ms, jittered), then surfaces as 502. Retried-then-succeeded sends return 200.
3. No new required env vars; all retry/metrics config has safe defaults. Existing `/send`, `/send/otp`, `/send/magic-link` behavior unchanged when retry budget = 0.
4. Test suite stays green; new tests cover retry budget exhaustion, retry-then-success, and `/metrics` content. Coverage delta ≥ 0%.
5. PyPI release published as `email-service` (or `smtp-email-service` if name is taken). `pip install email-service[http]` works without `git+`.

## Distribution Plan
- **Primary: PyPI.** Reserve name, configure trusted publishing via GitHub Actions OIDC (no long-lived tokens). Tag `v0.2.0` → workflow builds sdist + wheel → uploads. Add badge to README. This unblocks the README's `pip install` examples that currently rely on `git+`.
- **Secondary: GHCR container image.** GitHub Actions builds `ghcr.io/hwan96-ai/email-service:0.2.0` and `:latest` from the existing Dockerfile on tag push. Lets ops pull without a Python toolchain.
- **Tertiary: GitHub Releases** with autogenerated changelog from conventional commits. Source of truth for "what changed" — README links to it.
- No Homebrew, no Docker Hub mirror, no Helm chart in this round. YAGNI until someone asks.

## Next Steps
Concrete build order (each step independently shippable, each ends green):

1. **Retry primitive in `sender.py`** — add `max_retries: int = 0` and `retry_backoff_base: float = 0.1` to `SmtpConfig`. Wrap the `smtplib.SMTP(...)` block in a retry loop catching `smtplib.SMTPException` subclasses considered transient (`SMTPServerDisconnected`, `SMTPConnectError`, `SMTPResponseException` with 4xx). Jittered exponential backoff. Default `max_retries=0` preserves current behavior.
2. **Wire env vars** in `api.py::_build_sender_from_env`: `EMAIL_RETRY_MAX` (default 3 in HTTP mode), `EMAIL_RETRY_BACKOFF_BASE` (default 0.1). Library mode default stays 0.
3. **Metrics module** — new `email_service/metrics.py` with hand-rolled `Counter` and `Histogram` (no `prometheus_client` dep; ~50 LOC). Thread-safe via `threading.Lock`. Exposition function returns text.
4. **Instrument send paths** — record `email_send_total`, `email_send_duration_seconds`, `email_send_retries_total` in each of the three endpoints. Labels: `endpoint`, `outcome`.
5. **`GET /metrics` endpoint** — unauthenticated by default (standard Prom convention; document the "put it behind a reverse proxy" caveat in README, same as `/health`). Optional `METRICS_REQUIRE_AUTH=true` gate.
6. **Tests** — `test_sender_retry.py` (mock `smtplib.SMTP` raising then succeeding), `test_metrics.py` (counter increments, histogram bucket boundaries, exposition format parses), extend `test_api.py` for `/metrics` content.
7. **Docs** — README section "Metrics & Retry" with a Prometheus scrape config snippet and a Grafana panel JSON example. Update environment variable table.
8. **PyPI release pipeline** — `.github/workflows/release.yml` triggered on `v*` tag: build with `python -m build`, publish via `pypa/gh-action-pypi-publish` with OIDC. Reserve name first via manual one-time upload of `0.1.0`.
9. **GHCR image workflow** — extend the same release workflow to `docker/build-push-action` against existing `Dockerfile`. Tag with version and `latest`.
10. **Cut `v0.2.0`** — tag, let CI publish, update README install commands to drop `git+`.
