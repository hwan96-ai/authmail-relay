# SPEC

> 마지막 갱신: 2026-04-19 · 작성자: Planner Agent

## 1. 한 줄 요약
사용자의 백엔드 서비스가 HTTP 호출 한 번으로 이메일(일반/매직링크/OTP)을 발송할 수 있도록, 기존 email-service 패키지를 독립 HTTP 서비스로 감싼다.

## 2. 가정 (Assumptions)
- A1: 소비자는 사용자 본인이 운영하는 백엔드 서비스 1개. 외부 공개 API 아님(내부망 전제).
- A2: 인터페이스는 HTTP API 단독. MCP·CLI는 이번 범위 밖.
- A3: SMTP 자격증명(호스트/계정/비번/from)은 서버 시작 시 환경변수로 1회 로드. 요청 인자로는 절대 받지 않음.
- A4: 인증은 환경변수로 설정한 단일 API 키(공유 비밀). RBAC/멀티테넌시 없음.
- A5: 서비스는 단일 프로세스로 기동되는 독립 실행 파일 형태. 요청/응답은 JSON.
- A6: 내부에서 기존 `email_service` 패키지(SmtpSender, MagicLinkNotifier, OTPNotifier)를 그대로 사용. 발송 로직 재구현 금지.

## 3. 사용자 스토리
- 백엔드 개발자는 HTTP 요청 한 번으로 일반 HTML 이메일을 발송할 수 있다.
- 백엔드 개발자는 HTTP 요청 한 번으로 매직링크 이메일을 발송할 수 있다.
- 백엔드 개발자는 HTTP 요청 한 번으로 OTP 이메일을 발송할 수 있다.
- 서비스 운영자는 SMTP 설정과 API 키를 환경변수로만 주입한다(코드·요청에 비밀값 없음).
- 서비스 운영자는 서버 기동 시 설정 누락을 즉시 알 수 있다.

## 4. 완료 기준 (Acceptance Criteria)

- [ ] **AC-1**: 유효한 API 키로 일반 이메일(받는사람·제목·HTML 본문) 발송을 HTTP로 요청하면, 실제 SMTP 서버로 메일이 전송되고 성공 응답을 받는다.
- [ ] **AC-2**: 유효한 API 키로 매직링크 이메일(받는사람·사용자명·토큰) 발송을 요청하면, base_url+경로+토큰 쿼리 형태의 링크가 포함된 메일이 전송된다.
- [ ] **AC-3**: 유효한 API 키로 OTP 이메일(받는사람·사용자명·코드) 발송을 요청하면, OTP 코드가 포함된 메일이 전송된다.
- [ ] **AC-4**: API 키가 없거나 잘못된 요청은 인증 실패 응답(401)으로 거부되고, SMTP 호출이 일어나지 않는다.
- [ ] **AC-5**: 필수 필드(받는사람 등)가 누락된 요청은 검증 실패 응답(4xx)으로 거부된다.
- [ ] **AC-6**: SMTP 자격증명 환경변수 중 하나라도 빠진 채로 서버를 시작하면, 명시적 오류 메시지와 함께 기동이 즉시 실패한다(조용한 시작 금지).

### Edge Cases
- [ ] **AC-E1**: SMTP 서버 연결이 실패하면 예외로 터지지 않고, 서버 에러 응답(5xx 또는 정의된 실패 JSON)으로 반환된다.
- [ ] **AC-E2**: 받는사람/제목 필드에 CRLF(개행) 문자를 주입한 요청은 메일이 전송되지 않고 실패 응답을 받는다(기존 sender 가드 활용).

## 5. 스프린트

### Sprint 1 — HTTP 서버 뼈대 + 일반 발송 + 인증
- 다루는 AC: AC-1, AC-4, AC-5, AC-6
- 산출물: 서버 실행 후 curl로 일반 이메일 발송 엔드포인트 호출 → 실제 메일 도착(또는 모킹 테스트 통과). API 키 검증·환경변수 누락 시 기동 실패 동작.

### Sprint 2 — 매직링크 + OTP 엔드포인트
- 다루는 AC: AC-2, AC-3
- 산출물: 두 개 전용 엔드포인트가 동작. 기존 Notifier 재사용.

### Sprint 3 — 에러 경로 다듬기 + 회귀 테스트
- 다루는 AC: AC-E1, AC-E2
- 산출물: SMTP 실패/헤더 인젝션 시도에 대한 통합 테스트. 응답 포맷 일관성.

## 6. 범위 밖 (Out of Scope)
- MCP 서버 / CLI 인터페이스
- 다중 SMTP 계정(멀티테넌시) 및 요청별 자격증명 전달
- 발송 큐, 재시도, 레이트 리밋, 비동기 잡
- OAuth/JWT/RBAC 등 복잡한 인증
- 관리자 UI, 웹훅, 발송 로그 영속화
- 공개 인터넷 노출 전제(TLS 종단·WAF 등)
- Docker/Kubernetes 배포 파일 자동 생성
- TemplateNotifier 용 커스텀 템플릿 엔드포인트(1차에선 제외)

## 7. 열린 질문 (Open Questions)
- Q1: 새 HTTP 서비스 코드를 같은 저장소(예: `email_service` 패키지 내부 서브모듈) / 같은 저장소의 별도 패키지 / 별도 저장소 중 어디에 둘지. 기본 가정은 **같은 저장소·같은 패키지 내부 서브모듈**.
- Q2: 응답에 발송 추적용 식별자(message id 등)가 필요한지. 기본 가정은 **불필요**(단순 성공/실패).
